# AI提示词开发企业级规范 v1.0.0
适用于提示词企业级全栈项目，与现有TypeScript+Next.js全栈开发规范、以太坊开发规范、LangChain/LangGraph技术栈完全兼容对齐。所有标注「必须」的条款为强制执行项，特殊场景需AI技术委员会与安全合规委员会双审批后方可调整。

---

## 核心原则

1. **安全合规优先**：所有提示词开发以数据安全、系统安全、合规风控为最高优先级，杜绝提示词注入、越狱、敏感信息泄露、违规内容生成等风险；
2. **类型安全与标准化**：全链路与企业现有TS技术栈对齐，强制结构化输入输出，实现编译时+运行时双校验，杜绝格式断层导致的业务故障；
3. **可复现可审计**：所有提示词必须可稳定复现输出效果，全生命周期变更、调用、测试记录可追溯、可审计；
4. **业务确定性优先**：企业级提示词以输出确定性、稳定性为核心目标，优先保障业务规则落地，而非创意性发散；
5. **资产化管理**：所有提示词为企业核心数字资产，必须纳入统一的版本、权限、运维管理体系，禁止散落在业务代码、个人文档中。

## 技术栈强制要求

遵循唯一首选制，禁止使用未纳入清单的工具/框架：

| 开发方向             | 唯一首选技术栈                     | 禁止/不推荐替代方案              |
| -------------------- | ---------------------------------- | -------------------------------- |
| 提示词核心框架       | LangChain.js ChatPromptTemplate    | 硬编码字符串模板、自定义模板封装 |
| 输入输出类型校验     | Zod                                | Yup、Joi、无校验方案             |
| 提示词版本管理       | Git + 语义化版本                   | 在线文档、本地文件、无版本管控   |
| 提示词测试与可观测性 | LangSmith                          | 自定义测试脚本、无测试方案       |
| 提示词安全防护       | Lakera Guard + Zod输出校验         | 无防护方案、自定义过滤规则       |
| 结构化输出           | LangChain.js 结构化输出 + Zod      | 纯文本格式约束、无结构化方案     |
| 提示词编排           | LangGraph.js                       | 线性Chain、自定义流程封装        |
| 环境与配置管理       | Next.js 环境变量 + Vercel 环境隔离 | 硬编码密钥、环境混用             |
| 日志与审计           | Pino + Sentry                      | console.log、无日志方案          |

## 项目结构约定

```
prompts/
├── [business-module]/          # 按业务模块划分目录
│   ├── [prompt-name].prompt.ts # 提示词文件，使用.prompt.ts后缀
│   ├── [prompt-name].schema.ts # 输入输出Zod Schema定义
│   ├── [prompt-name].test.ts   # Vitest单元测试用例
│   └── [prompt-name].md        # 说明文档
├── templates/                  # 通用提示词模板库
└── utils/                      # 通用工具函数
```

### 命名规范

- 提示词文件：小写蛇形命名，后缀`.prompt.ts`，例如`rag_answer_generator.prompt.ts`
- Schema文件：与提示词同名，后缀`.schema.ts`
- 变量、类型：遵循TypeScript PascalCase/CamelCase企业规范

## 开发全流程规范

### 需求与架构设计
所有提示词开发必须先完成设计，禁止无设计直接编写：
1. 必须明确业务场景、核心目标、成功指标
2. 必须明确输入边界、输出要求、异常处理规则
3. 必须明确token限制、模型选型、调用频率约束
4. 必须职责单一，单个提示词仅负责单一业务职责，复杂流程通过LangGraph拆分为多个节点
5. 必须设计安全防护层，与企业安全规范对齐
6. 必须与现有TS技术栈兼容，复用现有类型定义

### 标准化提示词编写结构
所有提示词必须包含以下8个固定顺序模块，无内容标注「无特殊要求」，禁止省略：

1. **角色定位**：精准定义身份、专业领域、能力边界，与业务场景强绑定
2. **核心目标**：1-2句话明确唯一核心任务，禁止多目标混杂
3. **执行规则**：分点列出可落地、可校验的执行步骤，覆盖所有正常场景
4. **输入规范**：明确所有输入变量的含义、类型、约束，与Zod Schema完全一致
5. **输出规范**：明确输出格式结构，与Zod Schema完全一致，明确禁止内容
6. **约束与禁忌**：列出绝对禁止的行为、内容，包含合规安全要求
7. **异常处理规则**：明确所有异常场景的标准处理方式
8. **参考示例（Few-Shot）**：复杂场景提供2-3组覆盖正常/边界/异常的示例，简单场景标注「无特殊要求」

### 编写强制红线
- 禁止在提示词中硬编码API密钥、数据库地址、商业机密等敏感信息，所有敏感信息必须通过环境变量管理
- 禁止使用模糊、模棱两可、主观化的描述，所有规则必须可落地可校验
- 禁止在系统提示词中允许模型突破约束，禁止出现「除非用户要求」「特殊情况可例外」等表述
- 禁止无边界的泛化任务，必须明确能力边界，禁止允许模型执行未授权操作
- 禁止要求模型编造虚构信息，RAG场景必须明确仅使用提供的上下文信息
- 禁止使用纯自然语言约束输出格式，必须配合Zod Schema实现结构化强制约束

### 类型安全强制规范
- 所有输入必须定义TS类型+Zod校验Schema，输入先校验再传入模型，校验不通过直接拦截
- 所有输出必须定义Zod Schema，强制使用LangChain结构化输出能力
- 输出必须经过Zod二次校验，校验不通过触发异常处理，禁止流入下游
- Schema必须复用业务代码类型定义，禁止重复定义导致不一致
- 工具调用入参必须与Zod Schema完全一致，禁止调用未授权工具

## 版本与权限管理规范

### 版本管理
- 所有提示词统一存放于独立Git仓库，按业务模块分目录，禁止硬编码在业务代码中
- 遵循语义化版本规则：`主版本号.次版本号.补丁版本号`
  - 主版本：不兼容的架构/规则/目标变更
  - 次版本：功能新增、规则优化，向下兼容
  - 补丁版本：bug修复、文案优化，向下兼容
- 变更提交信息遵循Conventional Commits规范
- 每次变更必须附带测试报告，经过Code Review（1名AI工程师+1名业务负责人审批）方可合并
- 生产上线版本必须打Git Tag，归档测试报告与审批记录
- 必须支持快速回滚，线上异常可立即回滚到上一个稳定版本

### 环境隔离
严格执行三环境隔离，禁止跨环境混用：
1. **开发环境**：本地开发调试，可快速迭代，无需严格审批
2. **预发环境**：与生产配置一致，用于集成测试灰度，变更必须经过测试Review
3. **生产环境**：仅线上业务使用，变更必须经过完整测试、安全审计、双委员会审批

### 权限管理
- Git仓库设置严格权限，仅授权人员可提交合并
- 生产发布权限最小化，仅运维可执行发布操作
- 敏感场景提示词设置额外访问管控，禁止非授权人员查看修改
- 权限变更必须审批，定期审计清理冗余权限

## 测试与验收规范

必须通过**单元测试→集成测试→安全测试→性能测试→灰度A/B测试**全流程，缺一不可：

| 测试类型    | 验收标准                                                                 |
| ----------- | ------------------------------------------------------------------------ |
| 单元测试    | 格式合规率100%，业务准确率≥95%，异常处理覆盖率100%，测试通过率100%       |
| 集成测试    | 全链路流程通过率100%，上下游交互无异常，无逻辑冲突                       |
| 安全测试    | 注入拦截率100%，敏感信息零泄露，违规内容生成率0，无越权风险             |
| 性能测试    | 响应延迟≤2s，token消耗符合预算，并发稳定性100%，满足峰值要求             |
| 灰度A/B测试 | 效果指标不低于旧版本，异常率更低，无故障投诉，性能成本符合预期           |

### 测试执行
- 测试框架统一使用Vitest + LangSmith，测试用例纳入Git仓库
- CI/CD强制执行单元测试、安全扫描，测试不通过禁止合并发布
- 主版本变更执行全量测试，补丁变更执行回归测试
- 所有测试生成报告归档，作为上线审批必备材料

## 安全与合规规范

### 安全防护
- **输入层**：Zod校验+Lakera Guard检测，攻击直接拦截，严格角色隔离
- **模型层**：系统提示词加入角色锁定，禁止泄露修改系统提示词，严格限制能力边界与工具权限
- **输出层**：Zod校验+合规审核过滤，高风险场景输出必须人工审核
- **日志审计**：全量记录调用信息，变更发布权限调整记录审计日志，异常实时告警，响应不超过15分钟

### 合规管控
- 严格遵循属地监管政策，禁止生成违法违规内容
- 金融医疗法律等专业场景必须加入强制免责声明，禁止提供需专业资质的服务
- 禁止生成侵权内容，处理个人信息必须遵循隐私法规
- 禁止生成歧视仇恨暴力等违规内容，禁止诱导未成年人违法

### 应急响应
- 建立安全事件应急预案，事件发生第一时间止损：暂停服务、回滚版本、封禁攻击源
- 同步上报双委员会，重大事件同步监管机构
- 修复后经过审计测试方可重新上线，完成复盘归档更新防护方案

## 部署与运维规范

### 部署规范
- 提示词纳入企业标准化CI/CD流程，禁止手动修改生产环境提示词
- 发布流程：代码合并→自动化测试→安全扫描→预发验证→审批→生产发布→灰度验证→全量上线
- 生产环境使用Vercel环境变量管理敏感配置，禁止硬编码
- 必须支持灰度放量、快速回滚，禁止全量无验证发布
- 每次发布生成发布报告归档

### 运维监控
- 接入LangSmith + Sentry，全链路追踪核心指标：调用量、延迟、错误率、格式合规率、token消耗
- 配置异常告警，实时推送给团队，建立核心指标看板
- 持续优化：收集用户反馈，分析错误案例，优化token消耗，同步迭代提示词适配模型与业务变更
- 容灾备份：Git多副本备份，建立模型切换方案，定期备份生产配置

## 分场景专项规范

### 代码生成场景
- 必须严格遵循企业全栈开发规范、以太坊开发规范，禁止生成不符合规范的代码
- 必须包含安全校验：防SQL注入、防XSS、防重入、权限控制
- 必须包含完整类型定义、注释、异常处理
- 必须生成可直接编译执行的完整代码，禁止伪代码
- 禁止生成侵权、恶意代码

### RAG检索增强生成场景
- 必须仅使用提供的上下文信息回答，禁止编造内容
- 上下文不足时明确告知用户，禁止猜测
- 必须标注引用来源，便于溯源
- 禁止脱离上下文输出预训练内容，禁止出现「根据我的知识」表述

### 智能体/工具调用场景
- 单个提示词仅负责单一LangGraph节点决策，禁止覆盖全流程
- 严格约束工具调用规则、顺序、权限，禁止调用未授权工具
- 工具入参必须与Zod Schema一致，禁止非法参数
- 明确工具调用异常处理与重试降级方案
- 必须要求任务完成后输出结束标识，禁止无限循环调用

### 合规审核场景
- 所有规则必须可量化可校验，禁止模糊标准
- 必须输出结构化审核结果，与Zod Schema一致
- 明确边缘场景判定规则，禁止模棱两可结果
- 高风险内容必须触发人工审核，禁止模型直接终审
- 规则变更必须经过合规委员会审批，全量测试后方可上线

## 协作规范

- 所有变更必须经过Code Review，重点检查规范符合性、业务完整性、安全合规、测试覆盖、类型完整性
- 每个提示词必须配套文档，包含业务场景、核心目标、输入输出Schema、版本历史、测试报告、维护人，文档与代码同步更新
- 建立企业级提示词资产库，实现复用，禁止重复开发
- 开发人员必须通过规范考核方可参与开发
- 明确责任人制度，奖惩分明

# AI提示词开发企业级规范
*文档版本: v1.0.0*

---

## 核心原则

1. **安全合规优先**：所有提示词开发以数据安全、系统安全、合规风控为最高优先级，杜绝提示词注入、越狱、敏感信息泄露、违规内容生成等风险
2. **类型安全与标准化**：全链路与企业现有TS技术栈对齐，强制结构化输入输出，实现编译时+运行时双校验，杜绝格式断层导致的业务故障
3. **可复现可审计**：所有提示词必须可稳定复现输出效果，全生命周期变更、调用、测试记录可追溯、可审计
4. **业务确定性优先**：企业级提示词以输出确定性、稳定性为核心目标，优先保障业务规则落地，而非创意性发散
5. **资产化管理**：所有提示词为企业核心数字资产，必须纳入统一的版本、权限、运维管理体系，禁止散落在业务代码、个人文档中

## 技术栈限制（唯一首选制）

| 开发方向             | 唯一首选技术栈                     | 禁止/不推荐替代方案              |
| -------------------- | ---------------------------------- | -------------------------------- |
| 提示词核心框架       | LangChain.js ChatPromptTemplate    | 硬编码字符串模板、自定义模板封装 |
| 输入输出类型校验     | Zod                                | Yup、Joi、无校验方案             |
| 提示词版本管理       | Git + 语义化版本                   | 在线文档、本地文件、无版本管控   |
| 提示词测试与可观测性 | LangSmith                          | 自定义测试脚本、无测试方案       |
| 提示词安全防护       | Lakera Guard + Zod输出校验         | 无防护方案、自定义过滤规则       |
| 结构化输出           | LangChain.js 结构化输出 + Zod      | 纯文本格式约束、无结构化方案     |
| 提示词编排           | LangGraph.js                       | 线性Chain、自定义流程封装        |
| 环境与配置管理       | Next.js 环境变量 + Vercel 环境隔离 | 硬编码密钥、环境混用             |
| 日志与审计           | Pino + Sentry                      | console.log、无日志方案          |

## 项目结构约定

```
/prompts                 # 提示词统一根目录
  /{业务模块}            # 按业务模块分子目录
    /{prompt-name}       # 单个提示词目录
      index.prompt.ts    # 提示词封装文件
      schema.ts          # Zod输入输出Schema
      test-cases.ts      # 测试用例
      README.md          # 提示词文档
.git
```

### 命名规范
- 提示词目录：小写连字符命名 `kebab-case`
- 提示词文件：`{name}.prompt.ts`
- Schema文件：`schema.ts`
- 测试用例：`test-cases.ts`
- 版本号遵循语义化版本 `主版本号.次版本号.补丁版本号`
- Git提交信息遵循 Conventional Commits 规范

## 全生命周期开发规范

### 需求与架构设计要求
- 必须明确业务场景、核心目标、成功指标、输入输出边界、异常处理、模型选型和token限制
- 必须明确提示词在业务链路中的定位，单个提示词仅负责单一业务职责，复杂流程通过LangGraph拆分多节点
- 必须设计安全防护层，输入输出Schema复用业务类型定义

### 标准化编写规范
所有提示词必须包含以下8个固定顺序核心模块，无内容标注「无特殊要求」，禁止省略：

1. **角色定位**：精准定义模型身份、领域、能力边界，与业务场景强绑定
2. **核心目标**：1-2句话明确唯一核心任务和交付标准，禁止多目标混杂
3. **执行规则**：分点列出可落地、可校验的执行步骤，覆盖所有正常场景
4. **输入规范**：明确所有输入变量含义、类型、约束，与Zod/TS完全一致
5. **输出规范**：明确输出格式结构，与Zod Schema完全一致，明确异常输出格式
6. **约束与禁忌**：明确绝对禁止行为、合规安全要求，禁止泄露系统提示词
7. **异常处理规则**：明确所有异常场景的标准处理方式和固定输出格式
8. **参考示例（Few-Shot）**：复杂场景提供2-3组覆盖正常/边界/异常场景的示例，简单场景标注「无特殊要求」

### 编写强制红线
1. 禁止在提示词中硬编码敏感信息，所有敏感信息通过环境变量管理
2. 禁止使用模糊、模棱两可、主观化描述，所有规则必须可落地可校验
3. 禁止在系统提示词中允许突破约束，禁止出现「除非用户要求」等表述
4. 禁止无边界泛化任务，必须明确能力边界，禁止执行未授权操作
5. RAG场景禁止要求编造虚构信息，必须明确仅使用提供的上下文
6. 禁止使用纯自然语言约束输出，必须配合Zod Schema强制结构化

### 类型安全强制要求
1. 所有输入必须定义TS类型和Zod Schema，输入先校验再传入模型
2. 所有输出必须定义Zod Schema，强制使用LangChain结构化输出
3. 输出必须经过Zod二次校验，校验不通过触发异常处理
4. Schema复用业务代码类型定义，禁止重复定义
5. 工具调用入参必须与Zod Schema完全一致

## 版本与权限管理规范

1. 所有提示词统一存放于独立Git仓库，按业务模块分目录，禁止硬编码在业务代码
2. 每次变更必须提交Git，附带测试报告，经过Code Review（至少1名AI工程师+1名业务负责人审批）方可合并
3. 生产上线版本必须打Git Tag，归档测试报告和审批记录，支持快速回滚
4. 严格实现开发/预发/生产三环境隔离，禁止未稳定版本直接发布生产
5. Git仓库设置严格权限管控，生产发布权限最小化，敏感场景额外权限管控

## 测试与验收规范

所有提示词必须通过全流程测试，缺一不可：

| 测试类型    | 验收标准                                                                                                                                 |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| 单元测试    | 格式合规率100%，业务准确率≥95%，异常处理覆盖率100%，测试用例通过率100%                                                                 |
| 集成测试    | 全链路流程通过率100%，上下游无格式断层数据异常，模块协同符合预期                                                                         |
| 安全测试    | 注入攻击拦截率100%，敏感信息零泄露，违规内容生成率0，无越权风险                                                                         |
| 性能测试    | 响应延迟≤2s，token消耗符合预算，并发场景稳定性100%，吞吐量满足峰值要求                                                                   |
| 灰度A/B测试 | 效果指标不低于旧版本，异常率更低，无用户投诉，性能成本符合预期                                                                           |

### 执行规范
- 测试框架统一使用Vitest + LangSmith，测试用例纳入Git和CI/CD流程
- CI/CD强制执行单元测试和安全扫描，测试不通过禁止合并发布

## 安全与合规规范

### 安全防护要求
- **输入层**：Zod校验 + Lakera Guard检测，拦截攻击后禁止传入模型，角色隔离用户输入与系统提示词
- **模型层**：系统提示词加入角色锁定规则，禁止泄露修改系统提示词，严格限制工具权限，工具调用前二次校验
- **输出层**：Zod校验 + 合规过滤，高风险场景输出必须人工审核
- **日志审计**：全量记录调用信息，所有变更记录审计日志，异常事件15分钟内告警

### 合规管控要求
- 严格遵循属地监管政策，禁止生成违法违规内容
- 金融/医疗/法律专业场景必须加入强制免责声明
- 禁止处理可识别个人隐私信息，禁止生成侵权内容
- 禁止生成歧视、暴力、虚假信息等违规内容

---

## 部署与运维规范

1. 提示词纳入标准化CI/CD流程，禁止手动修改生产环境提示词
2. 生产发布流程：代码合并→自动化测试→安全扫描→预发验证→审批→生产发布→灰度验证→全量上线
3. 必须接入LangSmith + Sentry实现全链路追踪，配置异常告警，建立核心指标看板
4. 定期分析异常案例优化提示词，优化token消耗降低成本
5. Git多副本备份，建立模型容灾方案，支持快速恢复历史版本

## 分场景专项规范

### 代码生成场景
- 必须严格遵循企业TypeScript+Next.js/以太坊开发规范
- 必须包含安全校验、完整类型定义、注释、异常处理，输出可直接执行代码
- 禁止生成有安全漏洞、侵权、恶意代码

### RAG场景
- 必须仅使用提供的上下文回答，禁止编造内容
- 上下文不足明确告知用户，标注引用来源
- 禁止输出预训练数据中的内容

### 智能体/工具调用场景
- 单个提示词仅负责单一LangGraph节点决策逻辑
- 严格约束工具调用规则和权限，禁止调用未授权工具
- 入参与Zod Schema一致，明确异常处理，任务完成输出结束标识禁止无限循环

### 合规审核场景
- 规则必须可量化可校验，输出结构化结果与Zod一致
- 高风险内容必须触发人工审核，规则变更需合规委员会审批

---

## 协作规范

1. 所有变更必须经过Code Review，重点检查规范符合性、业务完整性、安全合规、测试覆盖、类型定义
2. 每个提示词必须配套完整文档，与代码同步更新
3. 沉淀企业级提示词资产库，实现复用避免重复开发
4. 开发人员必须通过规范考核方可参与开发

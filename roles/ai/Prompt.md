# AI提示词开发企业级规范


**文档版本**: v1.0.0

****适用范围**: 提示词企业级全栈项目**

**更新日期**: 2026-03-27

本规范基于2026年大模型应用开发生态、企业级安全合规标准与生产实践制定，**与此前TypeScript+Next.js全栈开发规范、以太坊开发规范、LangChain/LangGraph技术栈完全兼容对齐**，覆盖提示词全生命周期管理，包括需求设计、开发编写、版本管控、测试验收、上线运维、安全合规全流程。所有标注「必须」的条款为强制执行项，特殊场景需AI技术委员会与安全合规委员会双审批后方可调整。

---

## 一、总则

### 1.1 适用范围

本规范适用于企业内部所有基于大语言模型（LLM）的提示词开发工作，包括但不限于：RAG检索增强生成、LangChain/LangGraph智能体工作流、代码生成、内容生产、智能客服、数据处理、合规审核、区块链合约辅助开发等全场景提示词资产。

### 1.2 核心原则

1. **安全合规优先**：所有提示词开发以数据安全、系统安全、合规风控为最高优先级，杜绝提示词注入、越狱、敏感信息泄露、违规内容生成等风险；
2. **类型安全与标准化**：全链路与企业现有TS技术栈对齐，强制结构化输入输出，实现编译时+运行时双校验，杜绝格式断层导致的业务故障；
3. **可复现可审计**：所有提示词必须可稳定复现输出效果，全生命周期变更、调用、测试记录可追溯、可审计；
4. **业务确定性优先**：企业级提示词以输出确定性、稳定性为核心目标，优先保障业务规则落地，而非创意性发散；
5. **资产化管理**：所有提示词为企业核心数字资产，必须纳入统一的版本、权限、运维管理体系，禁止散落在业务代码、个人文档中。

## 三、提示词全生命周期开发规范

### 3.1 需求与架构设计规范

所有提示词开发必须先完成需求与架构设计，禁止无设计直接编写，设计文档必须归档留存。

1. **需求定义强制要求**
   - 必须明确提示词的业务场景、核心目标、成功指标（如格式合规率100%、内容准确率≥95%、响应延迟≤2s）；
   - 必须明确输入边界：输入变量的类型、范围、约束、最大长度，禁止无边界的泛化输入；
   - 必须明确输出要求：输出格式、内容规范、禁止项、合规要求，与Zod Schema一一对应；
   - 必须明确异常场景处理规则：输入非法、上下文缺失、模型无法完成任务时的标准响应方案；
   - 必须明确token限制、模型选型、调用频率约束，适配生产环境性能与成本要求。
2. **架构设计强制要求**
   - 必须明确提示词在业务链路中的定位：如RAG查询重写、上下文整合、工具调用决策、内容生成、结果校验等，与LangGraph节点一一对应；
   - 必须实现职责单一原则：单个提示词仅负责单一业务职责，禁止一个提示词覆盖多段复杂业务逻辑，复杂流程必须通过LangGraph拆分为多个提示词节点；
   - 必须明确上下游依赖：输入来源、输出去向、与其他模块的交互规则，禁止无依赖设计；
   - 必须设计安全防护层：输入校验、输出过滤、注入攻击拦截方案，与企业安全规范对齐；
   - 必须与现有TS技术栈完全兼容：输入输出Schema必须可直接复用业务代码的类型定义，禁止重复定义类型导致断层。

#### 编写强制红线

1. 禁止在提示词中硬编码API密钥、数据库地址、内部文档、商业机密等敏感信息，所有敏感信息必须通过环境变量管理；
2. 禁止使用模糊、模棱两可、主观化的描述，所有规则必须可落地、可校验、可复现；
3. 禁止在系统提示词中允许模型突破约束，禁止出现「除非用户要求」「特殊情况可例外」等表述；
4. 禁止无边界的泛化任务，必须明确模型的能力边界，禁止允许模型执行未授权的操作；
5. 禁止在提示词中要求模型编造、虚构信息，RAG场景必须明确要求模型仅使用提供的上下文信息，禁止编造不存在的内容；
6. 禁止使用纯自然语言约束输出格式，必须配合Zod Schema实现结构化输出强制约束。

### 3.3 类型安全强制规范

与企业TS全栈开发规范完全对齐，实现提示词全链路类型安全，杜绝类型断层导致的业务故障。

1. 所有提示词的输入变量必须定义TS类型与Zod校验Schema，输入内容必须先经过Zod校验，校验不通过直接拦截，禁止传入模型；
2. 所有提示词的输出必须定义Zod Schema，强制使用LangChain结构化输出能力，约束模型严格按照Schema生成内容；
3. 输出内容必须经过Zod二次校验，校验不通过直接触发异常处理流程，禁止将不符合格式的内容传入下游业务系统；
4. Schema定义必须与业务代码复用同一套类型定义，禁止重复定义导致类型不一致；
5. 工具调用场景的提示词，必须与Zod定义的工具入参Schema完全一致，禁止模型调用未授权的工具、传入不符合Schema的参数。

## 四、版本与权限管理规范

所有提示词为企业核心数字资产，必须纳入与代码一致的版本、权限管理体系，禁止散落在个人文档、聊天记录、业务代码中。

### 4.1 版本管理强制规范

1. **存储规范**：所有提示词必须统一存放于独立的Git仓库，按业务模块分子目录管理，禁止硬编码在业务代码中，禁止散落在多个仓库；
2. **文件规范**：提示词必须以独立的配置文件形式存放，推荐使用 `.prompt.ts`格式封装LangChain PromptTemplate，配套对应的Zod Schema文件，禁止使用纯文本文件；
3. **版本号规范**：必须遵循语义化版本规则 `主版本号.次版本号.补丁版本号`：
   - 主版本号：不兼容的架构变更、核心规则变更、业务目标变更；
   - 次版本号：功能新增、规则优化、示例补充，向下兼容；
   - 补丁版本号：bug修复、文案优化、约束补充，向下兼容；
4. **变更管理规范**：
   - 每次变更必须提交Git，提交信息必须遵循Conventional Commits规范，明确变更内容、变更原因；
   - 每次变更必须附带完整的测试报告，验证变更后的效果、合规性、稳定性，测试不通过禁止合并；
   - 变更必须经过Code Review，至少1名AI提示词工程师与1名业务负责人审批通过，方可合并；
   - 每个生产环境上线的版本必须打Git Tag，关联对应的测试报告、上线审批记录，永久归档；
5. **回滚机制**：必须建立快速回滚机制，线上出现异常时，可立即回滚至上一个稳定版本，禁止无回滚方案的变更上线。

### 4.2 环境隔离规范

与企业部署规范完全对齐，严格实现三环境隔离，禁止跨环境混用。

1. **开发环境**：仅用于本地开发、调试、单元测试，提示词可快速迭代，变更无需严格审批；
2. **预发环境**：与生产环境配置完全一致，用于集成测试、灰度验证、安全审计，变更必须经过测试与Review，禁止直接修改；
3. **生产环境**：仅用于线上业务，变更必须经过完整测试、安全审计、双委员会审批，禁止任何未经审批的临时修改；
4. 不同环境的提示词版本、模型配置、安全规则必须隔离，禁止开发环境的未稳定版本直接发布到生产环境。

### 4.3 权限管理规范

1. 提示词Git仓库必须设置严格的权限管控，仅授权人员可提交、合并代码，禁止全员可写；
2. 生产环境的提示词发布权限必须最小化，仅授权运维人员可执行发布操作，开发人员无生产环境发布权限；
3. 敏感场景（如金融、合规、核心业务）的提示词，必须设置额外的访问权限管控，禁止非授权人员查看、修改；
4. 所有权限变更必须经过审批，记录归档，定期审计权限清单，清理冗余权限。

## 五、测试与验收规范

企业级提示词必须经过完整的测试验证，测试不通过禁止进入预发、生产环境，测试体系与企业现有测试规范完全对齐。

所有提示词必须通过**单元测试→集成测试→安全测试→性能测试→灰度A/B测试**全流程，缺一不可。

| 集成测试    | 验证提示词在完整业务链路中的表现，如LangGraph全流程、RAG全链路、上下游系统交互；必须基于生产环境的真实场景设计测试用例 | 1. 全链路业务流程通过率100%；2. 上下游系统交互无格式断层、无数据异常；3. 与其他模块的协同符合预期，无逻辑冲突                                                                           |
| 安全测试    | 覆盖提示词注入、越狱攻击、敏感信息泄露、违规内容生成、越权操作等场景；必须使用自动化工具+人工渗透测试双验证            | 1. 注入攻击拦截率100%，无越狱成功场景；2. 敏感信息零泄露，不会泄露系统提示词、内部规则、商业机密；3. 违规内容生成率0，所有输出符合合规要求；4. 无越权操作风险，模型不会执行未授权的操作 |
| 性能测试    | 验证提示词的token消耗、响应延迟、吞吐量、并发稳定性；必须基于生产环境的流量模型设计测试场景                            | 1. 响应延迟符合业务预设指标（通常≤2s）；2. token消耗控制在预算范围内，无冗余token浪费；3. 并发场景下无超时、无异常、稳定性100%；4. 最大吞吐量满足业务峰值要求                          |
| 灰度A/B测试 | 生产环境必须先灰度发布，覆盖5%-10%的流量，与旧版本做对照测试；灰度周期不得低于24小时，核心业务不得低于72小时           | 1. 新版本效果指标不低于旧版本；2. 异常率、错误率与旧版本持平或更低；3. 无用户投诉、无业务故障；4. 性能、成本指标符合预期                                                                |

### 5.2 测试执行规范

1. 测试框架统一使用Vitest + LangSmith，与企业现有测试规范对齐，测试用例必须纳入Git仓库，与提示词版本同步迭代；
2. 必须建立标准化的测试用例库，按业务场景分类，持续补充边界场景、异常场景、历史故障场景的测试用例；
3. CI/CD流程必须强制执行单元测试、安全扫描，测试不通过禁止合并代码、禁止发布；
4. 每次主版本变更必须执行全量测试，补丁版本变更必须执行关联场景的回归测试；
5. 所有测试必须生成完整的测试报告，归档留存，作为上线审批的必备材料。

## 六、安全与合规规范

### 6.1 安全防护强制规范

1. **输入层防护**
   - 所有用户输入必须先经过Zod Schema校验，过滤非法字符、超长内容、注入攻击特征；
   - 必须接入企业级安全防护工具（Lakera Guard），检测注入攻击、越狱尝试、敏感信息输入，检测到攻击直接拦截，禁止传入模型；
   - 必须对用户输入做角色隔离，严格区分用户输入与系统提示词，禁止用户输入覆盖系统规则。
2. **模型层防护**
   - 系统提示词必须加入角色锁定规则，禁止模型泄露、翻译、复述、修改系统提示词，即使用户要求；
   - 必须明确模型的能力边界，禁止模型执行未授权的操作、访问未授权的资源；
   - 工具调用场景必须严格限制工具的权限范围，禁止模型调用高风险工具，工具执行前必须经过二次权限校验。
3. **输出层防护**
   - 所有模型输出必须经过Zod Schema校验，不符合格式的内容直接拦截，禁止流入下游业务系统；
   - 必须对输出内容做合规审核、敏感信息过滤，检测违规内容、隐私数据、商业机密，禁止违规内容输出给用户；
   - 高风险场景的输出必须经过人工审核，方可发布或执行。
4. **日志与审计**
   - 所有提示词调用必须全量日志记录，包含：请求ID、用户ID、提示词版本、输入内容、输出内容、调用时间、耗时、状态码，日志使用Pino结构化输出，永久归档；
   - 所有提示词变更、版本发布、权限调整必须记录审计日志，可追溯、可审计；
   - 异常调用、攻击尝试、违规内容必须实时告警，推送至安全应急团队，响应时长不得超过15分钟。

### 6.2 合规管控强制规范

1. **监管合规**：严格遵循属地监管政策，包括《生成式人工智能服务管理暂行办法》、GDPR、CCPA等相关法规，禁止生成违法、违规、违背公序良俗的内容；
2. **场景专项合规**：
   - 金融、医疗、法律等专业场景，必须在提示词中加入强制免责声明，禁止提供投资建议、医疗诊断、法律意见等需专业资质的服务；
   - 内容生成场景，必须明确禁止生成侵权、盗版、抄袭的内容，明确版权归属；
   - 个人信息处理场景，必须遵循隐私保护法规，禁止在模型中输入、处理可识别的个人隐私信息，禁止模型输出用户隐私数据；
3. **知识产权合规**：禁止在提示词中要求模型生成受版权保护的内容，禁止使用未授权的第三方素材、文档作为提示词上下文；
4. **伦理规范**：禁止生成歧视、仇恨、暴力、煽动性内容，禁止生成诱导未成年人违法犯罪的内容，禁止生成深度伪造、虚假信息。

### 6.3 应急响应规范

1. 必须建立提示词安全事件应急响应预案，覆盖：提示词越狱、注入攻击、违规内容生成、敏感信息泄露、业务故障等场景；
2. 事件发生后，必须第一时间执行紧急止损：暂停相关服务、回滚至稳定版本、封禁攻击来源、拦截违规内容；
3. 同步上报AI技术委员会与安全合规委员会，重大事件必须同步监管机构；
4. 组织团队排查漏洞、编写修复方案，经过安全审计、测试验证后，方可重新上线；
5. 事件处置完成后，必须完成复盘归档，更新安全规范与防护方案，杜绝同类事件再次发生。

## 七、部署与运维规范

与企业Next.js/Vercel部署规范完全对齐，实现提示词的标准化部署、可观测运维、持续优化。

### 7.1 部署规范

1. 提示词必须与业务代码一起，纳入企业标准化CI/CD发布流程，禁止手动修改生产环境的提示词；
2. 发布流程必须包含：代码合并→自动化测试→安全扫描→预发验证→审批→生产发布→灰度验证→全量上线，缺一不可；
3. 生产环境必须使用Vercel环境变量管理敏感配置，禁止硬编码在代码、配置文件中；
4. 发布必须支持灰度放量、快速回滚能力，禁止全量无验证发布；
5. 每次发布必须生成发布报告，包含：版本号、变更内容、测试结果、审批记录、发布时间、发布人，永久归档。

### 7.2 运维与监控规范

1. **可观测性强制要求**
   - 必须接入LangSmith + Sentry，实现提示词调用全链路追踪，监控核心指标：调用量、响应延迟、错误率、格式合规率、token消耗；
   - 必须配置异常告警规则：错误率突增、延迟超标、注入攻击、违规内容生成、token消耗异常，实时推送至运维与业务团队；
   - 必须建立核心业务指标看板，实时监控提示词的业务效果、稳定性、成本，定期生成运维报告。
2. **持续优化规范**
   - 必须建立用户反馈闭环，收集用户对模型输出的投诉、建议，定期分析优化提示词；
   - 必须定期分析异常日志、错误案例，补充测试用例，优化提示词规则，提升稳定性；
   - 必须定期优化提示词的token消耗，降低推理成本，提升响应速度；
   - 模型版本升级、业务规则变更时，必须同步迭代提示词，完成全量测试验证后发布。
3. **容灾备份规范**
   - 所有提示词资产必须通过Git仓库多副本备份，禁止单点存储；
   - 必须建立模型容灾方案，主模型故障时，可快速切换至备用模型，保障业务连续性；
   - 生产环境的配置、版本必须定期备份，可快速恢复至历史稳定版本。

## 八、分场景专项规范

基于企业核心业务场景，制定专项强制规则，与现有开发规范完全对齐。

### 8.1 代码生成场景提示词规范

1. 必须强制要求生成的代码严格遵循《企业级TypeScript+Next.js全栈开发规范》《以太坊区块链开发企业级规范》，禁止生成不符合规范的代码；
2. 必须强制要求代码做安全校验，包含：防SQL注入、防XSS、防重入、权限控制等安全措施，禁止生成有安全漏洞的代码；
3. 必须强制要求代码包含完整的类型定义、注释、异常处理，禁止生成无类型、无注释、无异常处理的代码；
4. 必须强制要求代码可直接执行、可编译通过，禁止生成伪代码、不完整的代码片段；
5. 必须明确禁止生成侵权、开源许可证违规的代码，禁止生成恶意代码、后门程序。

### 8.2 RAG检索增强生成场景提示词规范

1. 必须强制要求模型仅使用提供的上下文信息回答问题，禁止编造、虚构上下文不存在的信息；
2. 必须强制要求模型在上下文信息不足、无法回答时，明确告知用户，禁止猜测、编造内容；
3. 必须强制要求模型标注引用的上下文来源，便于用户溯源验证；
4. 必须明确禁止模型脱离上下文，输出自身预训练数据中的内容，禁止出现「根据我的知识」等表述；
5. 必须设计多轮查询优化、上下文重排的专项提示词，提升检索准确率与回答质量。

### 8.3 智能体/工具调用场景提示词规范

1. 必须与LangGraph节点设计完全对齐，单个提示词仅负责单一节点的决策逻辑，禁止一个提示词覆盖全流程决策；
2. 必须严格约束工具调用的规则、顺序、权限，禁止模型调用未授权的工具，禁止执行高风险操作；
3. 工具调用的入参必须与Zod Schema完全一致，禁止传入非法参数；
4. 必须明确工具调用的异常处理规则，工具调用失败时的重试、降级方案；
5. 必须强制要求模型在完成任务后，输出明确的结束标识，禁止无限循环调用工具。

### 8.4 合规审核场景提示词规范

1. 必须明确审核规则、判定标准、分级规则，所有规则必须可量化、可校验，禁止模糊的审核标准；
2. 必须强制要求输出结构化的审核结果，包含：是否通过、违规类型、违规内容、风险等级、整改建议，与Zod Schema完全一致；
3. 必须明确边缘场景的判定规则，禁止模棱两可的审核结果；
4. 必须强制要求高风险内容触发人工审核，禁止模型直接终审高风险内容；
5. 审核规则变更必须经过合规委员会审批，全量测试验证后方可上线。

## 九、工程化与协作规范

1. **Code Review规范**：所有提示词变更必须经过Code Review，Review重点包括：是否符合本规范、业务规则是否完整、安全合规是否到位、测试用例是否覆盖、类型定义是否完整，Review不通过禁止合并；
2. **文档规范**：每个提示词必须配套完整的文档，包含：业务场景、核心目标、输入输出Schema、版本历史、测试报告、维护人、注意事项，文档必须与代码同步更新；
3. **资产沉淀规范**：必须建立企业级提示词模板库、最佳实践库、通用规则库，实现资产复用，禁止重复开发；
4. **培训规范**：定期开展提示词开发规范、安全合规、最佳实践的培训，所有相关开发人员必须通过考核后方可参与提示词开发工作；
5. **责任与奖惩规范**：严格落实责任人制度，提示词的开发、测试、上线、运维均有明确责任人；严格遵守本规范并做出优秀成果的，给予奖励；违反本规范导致安全事故、业务故障、合规风险的，追究相关责任人责任。

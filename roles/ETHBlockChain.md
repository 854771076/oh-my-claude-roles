# 以太坊区块链企业级开发规范

**文档版本**: v1.0.0

****适用范围**: 区块链企业级全栈项目**

**更新日期**: 2026-03-27
**核心原则**: 能异步都异步、有框架用框架不手写重复代码、强类型安全、可测试、可维护、高性能

本规范基于2026年以太坊最新生态、行业安全标准与企业级生产实践制定，**与此前TypeScript+Next.js全栈开发规范完全兼容对齐**，覆盖智能合约开发、全栈DApp构建、安全管控、测试部署、运维合规全生命周期，所有标注「必须」的条款为强制执行项，特殊场景需区块链技术委员会与安全委员会双审批后方可调整。

---

## 一、总则

### 1.1 适用范围

本规范适用于企业内部所有以太坊生态相关开发工作，包括但不限于：智能合约开发、EVM兼容链开发、去中心化应用（DApp）全栈开发、链上协议开发、链上数据服务、账户抽象/钱包相关开发。

### 1.2 核心原则

1. **安全优先**：所有开发动作以链上安全为最高优先级，杜绝任何可预见的安全风险；
2. **类型安全**：全链路优先使用TypeScript，与现有全栈技术栈无缝对齐，杜绝类型断层；
3. **标准化合规**：严格遵循以太坊EIP标准、行业最佳实践与属地监管合规要求；
4. **可审计可追溯**：所有代码、部署、升级操作必须可审计、可追溯、可回滚；
5. **生产级验证**：所有上线主网的代码必须经过全量测试、多轮安全审计与生产环境验证。

## 三、智能合约开发规范

### 3.1 基础配置规范

1. **版本锁定**：必须在合约文件首行锁定Solidity编译器版本，禁止使用浮动版本（`^`），示例：`pragma solidity 0.8.25;`
2. **许可证声明**：必须在首行添加SPDX许可证声明，与项目开源/闭源属性匹配，禁止无许可证声明，示例：`// SPDX-License-Identifier: MIT`
3. **编译器配置**：
   - 必须开启优化器，默认优化层级为 `200`，生产环境禁止关闭优化器；
   - 必须指定EVM版本为最新稳定版（如 `cancun`），禁止使用已淘汰的EVM版本；
   - 禁止启用未经验证的实验性特性。
4. **目录结构**：必须遵循Foundry标准目录结构，禁止随意调整：
   ```
   ├── src/          # 智能合约源码，按业务模块分子目录
   ├── test/         # 测试用例，单元/集成/Fuzz测试分离
   ├── script/       # 部署/升级/运维脚本
   ├── lib/          # 依赖库（如OpenZeppelin）
   ├── audit/        # 审计报告归档
   ├── doc/          # 合约NatSpec文档与架构设计
   ```

### 3.2 命名规范（与TS全栈规范对齐）

| 代码元素          | 命名规则                     | 示例                                                | 强制要求                                            |
| ----------------- | ---------------------------- | --------------------------------------------------- | --------------------------------------------------- |
| 合约/库/接口      | PascalCase                   | `LendingPool`, `IERC20`                         | 接口必须以 `I`为前缀，禁止缩写                    |
| 函数/方法         | camelCase                    | `deposit`, `getUserBalance`                     | 外部函数必须以动词开头，明确表达业务含义            |
| 状态变量          | camelCase                    | `totalSupply`, `owner`                          | 必须显式声明可见性，禁止默认可见性                  |
| 常量/不可变变量   | UPPER_SNAKE_CASE             | `MAX_DEPOSIT_AMOUNT`, `DECIMALS`                | `immutable`变量必须在构造函数赋值，禁止运行时修改 |
| 修饰器            | camelCase                    | `onlyOwner`, `nonReentrant`                     | 必须以权限/状态校验为核心，禁止包含业务逻辑         |
| 事件              | PascalCase，以动词过去式结尾 | `DepositCompleted`, `OwnershipTransferred`      | 必须包含核心业务参数与indexed索引字段，便于链上检索 |
| 自定义错误        | PascalCase，以Error为后缀    | `InsufficientBalanceError`, `UnauthorizedError` | 禁止使用require+字符串错误，统一使用自定义错误      |
| 函数参数/局部变量 | camelCase                    | `userAddress`, `amount`                         | 禁止单字母变量名，除循环变量 `i`外                |

### 3.3 核心代码开发规范

1. **可见性控制**

   - 必须为所有状态变量、函数显式声明可见性（`external`/`public`/`internal`/`private`），禁止使用默认可见性；
   - 仅外部调用的函数优先使用 `external`，而非 `public`，节省gas；
   - 核心业务逻辑必须封装在 `internal`函数中，外部函数仅做参数校验与权限控制。
2. **错误处理规范**

   - 强制使用**自定义错误**替代 `require`/`revert`字符串，降低gas成本，提升错误可读性；
   - 必须遵循「先校验、后修改、再交互」的Checks-Effects-Interactions模式，这是防重入的核心强制规则；
   - 校验逻辑必须前置，所有参数、权限、状态校验必须在函数最顶部完成，校验不通过立即终止执行。
3. **状态管理规范**

   - 禁止在循环中操作存储变量，必须先读取到内存中处理，完成后一次性写入存储；
   - 只读函数参数优先使用 `calldata`，而非 `memory`，降低gas成本；
   - 存储变量必须合理打包，利用EVM存储槽特性，将多个小类型变量打包到同一个槽中，节省gas；
   - 可升级合约必须预留存储间隙，严格遵循OpenZeppelin可升级合约规范，禁止存储槽冲突。
4. **安全防护强制规范**

   - 所有涉及资金转出、外部合约调用的函数，必须使用OpenZeppelin `ReentrancyGuard` 防重入锁，禁止自定义防重入逻辑；
   - 权限控制必须使用 `Ownable2Step`（两步所有权转移），禁止使用旧版单步 `Ownable`；多角色权限必须使用 `AccessControl`，禁止硬编码地址权限；
   - 禁止使用 `tx.origin`做权限校验，杜绝钓鱼攻击风险；
   - 禁止使用 `selfdestruct`，兼容上海升级后EVM行为变更；
   - 禁止使用 `block.timestamp`、`block.number`、`blockhash`作为随机数来源，随机数必须使用Chainlink VRF；
   - 价格预言机必须使用Chainlink Price Feed，禁止直接使用DEX现货价格、Uniswap TWAP（无多重防护场景），杜绝价格操纵风险；
   - 禁止无限制的循环遍历，必须设置循环上限，杜绝DOS攻击风险；
   - 外部合约调用必须使用低级别调用，并严格校验返回值，禁止信任未审计的外部合约；
   - 禁止在修饰器中执行外部合约调用，杜绝不可预见的安全风险。
5. **标准兼容规范**

   - ERC20/ERC721/ERC1155等代币标准，必须使用OpenZeppelin官方实现，禁止自定义修改核心逻辑；
   - 签名验证必须严格遵循EIP-712结构化签名标准，禁止自定义签名方案，必须加入链ID、合约地址、nonce防重放；
   - 账户抽象相关开发必须遵循EIP-4337标准，禁止自定义账户抽象方案。
6. **注释规范**

   - 所有合约、接口、函数、事件、错误、状态变量必须添加**NatSpec注释**，包含 `@notice`、`@dev`、`@param`、`@return`标签；
   - 复杂业务逻辑、非常规操作、安全特殊处理必须添加详细行内注释，说明设计原因与风险点；
   - 必须基于NatSpec自动生成开发者文档，归档至 `doc/`目录。

## 四、全栈DApp开发规范（与TS+Next.js规范完全兼容）

### 4.1 前端开发规范（Next.js+TS）

1. **合约交互规范**

   - 强制使用**Wagmi v2+ + Viem**实现合约交互，禁止直接使用Viem底层API封装，禁止使用Ethers.js/Web3.js；
   - 钱包集成必须使用RainbowKit + WalletConnect，支持主流钱包，禁止自定义钱包连接逻辑，禁止硬编码钱包地址；
   - 合约ABI必须通过TypeChain生成TS类型，禁止手动编写ABI类型定义，实现全链路类型安全；
   - 合约调用必须做前置参数校验，使用Zod定义校验Schema，与此前全栈规范对齐，禁止非法参数上链；
   - 客户端仅做合约调用的交互层，所有业务逻辑、权限校验必须在链上合约中实现，禁止仅在客户端做权限控制。
2. **状态与数据规范**

   - 链上数据优先使用Wagmi内置的缓存机制，配合TanStack Query做客户端状态管理，与此前全栈规范对齐；
   - 批量链上数据查询必须使用Ponder索引器，禁止直接通过RPC `eth_getLogs`轮询，杜绝RPC限流与性能问题；
   - 链上事件监听必须使用Viem `watchContractEvent`，禁止轮询区块数据；
   - 禁止在链上存储用户隐私数据，所有隐私数据必须存储在链下加密数据库，链上仅存储数据哈希用于校验。
3. **安全规范**

   - 禁止在前端代码、Git仓库、客户端环境变量中暴露私钥、助记词、API密钥等敏感信息；
   - 敏感操作（转账、授权、合约升级）必须在前端添加二次确认，提示用户风险与gas成本；
   - 必须校验链ID与网络匹配性，禁止用户在错误网络中执行合约调用；
   - 签名操作必须明确告知用户签名内容与风险，禁止盲签，禁止签名无限期授权。

### 4.2 后端服务开发规范（Node.js+TS）

1. **链上交互规范**

   - 强制使用Viem实现链上交互，全链路TS类型安全，与前端类型复用；
   - 服务端私钥必须使用HSM/云KMS（AWS KMS/GCP KMS）管理，禁止硬编码在代码、环境变量、配置文件中；
   - 主网敏感操作必须通过Gnosis Safe多签执行，禁止使用单EOA账户直接执行；
   - RPC节点必须使用付费企业级服务（Alchemy/Infura），做多节点冗余配置，禁止使用公共RPC节点；
   - 必须实现RPC请求限流、缓存、重试机制，杜绝节点限流与单点故障。
2. **业务与安全规范**

   - 所有链上交易必须做全量日志记录（Pino），包含交易哈希、发起地址、区块高度、业务参数，可追溯可审计；
   - 交易执行必须实现幂等性，杜绝重复交易、重复扣款风险；
   - 签名验证必须在服务端严格校验，包含签名者地址、nonce、过期时间，杜绝重放攻击；
   - 错误追踪统一使用Sentry，全链路覆盖前端、服务端、链上交易异常，与此前全栈规范对齐。

## 五、测试规范

### 5.1 测试体系强制要求

所有智能合约必须通过**单元测试→集成测试→Fuzz测试→不变量测试→主网分叉测试→E2E端到端测试**全流程，缺一不可，测试不通过禁止进入审计与上线流程。

### 5.2 智能合约测试规范

1. **单元测试**

   - 测试框架统一使用Foundry，测试用例与合约文件一一对应；
   - 核心业务逻辑、权限控制、参数校验、异常场景必须100%全覆盖，代码行覆盖率必须达到100%，分支覆盖率≥95%；
   - 必须覆盖所有正常业务路径、异常回滚路径、边界条件，禁止仅测试正常场景。
2. **集成测试**

   - 必须测试合约与外部合约的交互逻辑（如DEX、预言机、其他协议）；
   - 必须覆盖完整业务流程，如借贷的存款-借款-还款-清算全链路；
   - 必须测试多合约协同、权限流转、升级逻辑的完整性。
3. **Fuzz与不变量测试（强制核心项）**

   - 核心业务逻辑必须编写Fuzz测试用例，覆盖所有输入边界，自动生成随机参数测试；
   - 必须编写不变量测试，定义协议核心不变量（如「用户存款总额≥可提现总额」「抵押率始终≥清算阈值」），通过长时间Fuzz验证不变量永不突破；
   - 不变量测试运行时长不得低于24小时，主网上线前必须完成百万级以上的Fuzz轮次。
4. **主网分叉测试**

   - 必须基于以太坊主网最新区块高度做分叉测试，模拟与主网已部署协议的真实交互；
   - 必须测试极端场景（如大额资金操作、全量用户清算、协议暂停与恢复）；
   - 必须在分叉环境中完成完整的部署、初始化、业务操作、升级全流程模拟。

### 5.3 全栈DApp测试规范

1. **单元/集成测试**：统一使用Vitest，与此前全栈规范对齐，核心业务逻辑、工具函数、hooks覆盖率≥80%；
2. **E2E端到端测试**：统一使用Playwright，覆盖核心用户流程（钱包连接、存款、转账、合约交互），测试环境使用Anvil本地测试节点，禁止使用测试网做E2E测试；
3. **API Mock**：统一使用MSW，与此前全栈规范对齐，禁止修改业务代码做Mock；
4. **CI/CD强制卡点**：所有测试必须在CI流程中自动执行，测试失败禁止合并代码、禁止部署。

## 六、安全审计与应急响应规范

### 6.1 安全审计强制流程

1. **前置内部审计**：代码提交第三方审计前，必须完成内部安全扫描：
   - 静态代码分析：Slither + Aderyn全量扫描，必须修复所有高危、中危漏洞，低危漏洞必须说明不修复的原因与风险控制方案；
   - 全量测试用例执行，100%通过；
   - 内部安全团队代码Review，至少2名区块链安全专家签字确认。
2. **第三方审计**：主网上线前必须经过至少1家顶级安全公司的全量审计，核心协议必须经过2家及以上顶级安全公司交叉审计（可选审计机构：OpenZeppelin、Trail of Bits、CertiK、SlowMist）；
3. **审计问题修复**：必须100%修复所有高危、中危漏洞，修复后必须经过审计方复核确认；
4. **审计归档**：所有审计报告、修复记录、复核报告必须归档至 `audit/`目录，永久留存；
5. **漏洞赏金**：主网上线后必须在Immunefi平台开启漏洞赏金计划，设置合理的赏金梯度，覆盖白帽黑客漏洞提交渠道。

### 6.2 应急响应规范

1. **监控告警**：必须通过OpenZeppelin Defender配置全量监控，覆盖：
   - 合约异常交易、大额资金转账；
   - 权限操作、合约升级、暂停/恢复操作；
   - 预言机价格异常、清算阈值突破等核心业务风险；
   - 告警必须实时推送至应急响应团队，响应时长不得超过15分钟。
2. **紧急处置流程**
   1. 漏洞确认：安全团队第一时间确认漏洞等级、影响范围、可利用性；
   2. 紧急止损：通过合约紧急暂停机制暂停核心业务，冻结风险资金，禁止漏洞利用；
   3. 事件上报：同步上报技术委员会与安全委员会，重大漏洞必须同步监管机构；
   4. 漏洞修复：开发团队编写修复方案，经过内部安全审计、第三方复核；
   5. 修复上线：通过多签+时间锁完成合约升级/修复，恢复业务；
   6. 公告披露：向用户同步事件详情、影响范围、处置方案与后续防护措施；
   7. 复盘归档：完成事件复盘，更新安全规范与防护方案，永久归档事件记录。
3. **强制防护机制**：所有主网合约必须内置紧急暂停机制，暂停权限必须由多签钱包控制，禁止单地址持有暂停权限。

## 七、部署与运维规范

### 7.1 部署规范

1. **部署环境隔离**：严格隔离三个环境，禁止跨环境混用资源：
   - 开发环境：Anvil本地节点，仅用于本地开发测试；
   - 测试环境：以太坊测试网（Sepolia/Holesky），用于测试网部署、集成测试、外部审计前验证；
   - 生产环境：以太坊主网，仅用于经过全量测试、审计的正式版本部署。
2. **部署脚本规范**
   - 必须使用Foundry Script编写部署脚本，脚本必须可重复执行、可审计、可追溯；
   - 必须使用CREATE2确定性部署，实现多链地址统一，便于验证与管理；
   - 部署脚本必须包含完整的合约初始化、权限配置、参数设置逻辑，禁止部署后手动修改配置；
   - 所有部署操作必须生成完整的部署报告，包含：合约名称、部署地址、交易哈希、区块高度、构造函数参数、部署人、部署时间，永久归档。
3. **部署权限规范**
   - 测试网部署可使用EOA账户，私钥必须严格隔离；
   - 主网部署必须使用Gnosis Safe多签钱包，禁止使用单EOA账户直接部署；
   - 部署后必须将合约所有权、管理员权限转移至多签钱包，禁止部署账户保留任何核心权限；
   - 部署完成后必须在Etherscan完成合约源码开源验证，验证内容必须与Git仓库代码完全一致。
4. **合约升级规范**
   - 可升级合约必须使用OpenZeppelin UUPS/透明代理模式，禁止自定义可升级方案；
   - 升级前必须完成全量测试、安全审计，验证存储兼容性，禁止存储槽冲突；
   - 升级操作必须通过多签钱包执行，必须设置时间锁（TimeLock），锁定期不得低于48小时，给用户预留风险应对时间；
   - 升级前必须发布公告，向用户同步升级内容、影响范围、时间安排；
   - 升级完成后必须重新验证合约源码，生成升级报告，永久归档。

### 7.2 节点与运维规范

1. **RPC节点规范**
   - 生产环境必须使用企业级付费RPC服务（Alchemy/Infura），配置至少2个不同服务商的节点做冗余，禁止使用公共RPC节点；
   - 必须配置RPC请求缓存、限流、熔断、重试机制，杜绝节点故障导致的服务不可用；
   - 历史数据查询必须使用归档节点，常规业务使用全节点，降低成本与请求延迟。
2. **链上运维规范**
   - 必须使用OpenZeppelin Defender做自动化运维，实现定时任务、余额监控、gas费优化、自动清算等自动化操作；
   - 必须建立gas费管理机制，极端gas价格场景下的交易策略，杜绝交易卡住、gas费过高的问题；
   - 所有链上运维操作必须通过多签执行，必须做全量日志记录，可审计可追溯；
   - 必须定期跟进以太坊网络升级，提前适配新的EIP与EVM特性，杜绝网络升级导致的合约故障。

## 八、合规与工程化规范

### 8.1 合规强制规范

1. **监管合规**：严格遵循属地监管政策，涉及法币兑换、稳定币交易、金融衍生品的业务，必须完成合规资质备案，落实KYC/AML要求；
2. **数据合规**：严格遵循GDPR/CCPA等数据隐私法规，禁止在链上存储用户隐私数据，用户数据必须加密存储，提供用户数据删除、导出功能；
3. **知识产权合规**：严格遵循开源许可证要求，使用OpenZeppelin等开源库必须保留许可证声明，禁止侵权使用第三方代码；
4. **税务合规**：必须完整留存所有链上交易记录，配合税务申报要求，建立完善的链上账务体系。

### 8.2 工程化与协作规范（与TS全栈规范对齐）

1. **Git管理规范**
   - 分支管理：`main`（主分支，对应主网正式版本）、`develop`（开发分支）、`feature/*`（功能分支）、`hotfix/*`（紧急修复分支）、`release/*`（发布分支）；
   - 提交规范：必须遵循Conventional Commits规范，提交信息必须明确表达业务内容，禁止无意义提交；
   - 代码合并：必须经过至少2名资深开发者Review，其中1名必须是区块链安全专家，CI测试全量通过，禁止直接推送到主分支。
2. **版本管理规范**
   - 合约版本必须遵循语义化版本（SemVer）：`主版本号.次版本号.补丁版本号`；
   - 不兼容升级、核心业务变更必须升级主版本号，新增功能升级次版本号，bug修复升级补丁版本号；
   - 每个主网发布版本必须打Git Tag，关联对应的审计报告、部署报告、更新日志。
3. **文档规范**
   - 必须维护完整的项目文档，包含：架构设计文档、合约技术文档、安全设计文档、部署文档、用户操作手册、应急响应预案；
   - 合约必须通过NatSpec生成完整的开发者文档，每个版本更新必须同步更新文档；
   - 必须定期开展安全培训与技术分享，跟进以太坊生态最新安全漏洞、技术升级与行业最佳实践。

本规范自发布之日起强制执行，所有以太坊相关开发工作必须严格遵循本规范，如有违反将追究相关责任人责任。
